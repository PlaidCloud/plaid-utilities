# Changelog

## Unreleased

- Fix `get_select_query` crashing with `TypeError: unsupported operand type(s) for -: 'NoneType' and 'int'` when a config carries a row-limit key that is present but null. `fill_in` fell back to its documented default only when the key was *absent* â€” `dict.get(key, default)` returns the stored `None` when the key is present â€” so `{'use_target_slicer': True, 'limit_target_end': None}` reached `limit_target_end - limit_target_start`. It is reachable from the Table Explorer: clearing or typing non-numeric text into the **Max Rows** box while the Max Rows checkbox is on sends `limit_rows` as JSON `null` (the client does `parseInt(...)`, and `NaN` serialises to `null`), which the server copies straight into `limit_target_end`. The grid preview showed a misleading "Unable to open previewâ€¦" alert, and **Download â†’ Limited Row Count** and **Save as View** returned a 500. All nine `fill_in`-guarded params (`aggregate`, `having`, `use_target_slicer`, `limit_target_start`, `limit_target_end`, `distinct`, `count`, `disable_variables`, `aggregation_type`) now treat a present-but-null config value as not supplied, matching how the function already treats an explicit `None` argument. Only the two limit params change behaviour: six of the remaining seven are consumed solely in truthiness tests, where `None` and the `False`/`None` default are interchangeable, and `aggregation_type` is unaffected because its signature default is `'group'` rather than `None`, so its config lookup is reached only when a caller passes the argument as `None` explicitly. A null limit now yields the same `LIMIT 0 OFFSET 0` as omitting the key entirely; callers are expected not to send a slicer with no row count (PlaidCloud's table explorer now clamps that case before it reaches this function) ([@inviscid](https://github.com/inviscid)).
- Fix expression validation rejecting `is`, `is not` and `not`, which broke saved calculated-field, filter and rule expressions that had run for years (e.g. a lookup step failing with "Is is not permitted in an expression."). The three operators were omitted from the expression allowlist in error; they add no escape surface and are now accepted again. One family is deliberately still rejected: `not (...)` wrapped around a column comparison (`==`, `!=`, `is`, `is not`, `in`, `not in`) does not build a SQL NOT â€” it collapses to a constant and silently drops the comparison from the query â€” so it now fails with a message naming what does work (`!=`, `is_(None)`, `notin_([...])` for a single comparison; `not_(and_(...))`/`not_(or_(...))` when the comparisons are joined by `and`/`or`). A `not (...)` around a comparison of pure constants (no column involved, e.g. `not ('ABC' == 'CORP')`) is correct and is left alone. Note that on a column `x is None` is a Python identity test that is always false â€” use `x.is_(None)` for a real null test (sc-23186) ([@simozzy](https://github.com/simozzy)).
- Fix expression validation rejecting columns whose name begins with an underscore, which broke steps that reference them (e.g. "Access to attribute '_MajorAccountFlag' is not permitted."). Such a column is reachable as `table._MyColumn` again, while a name that merely looks like one but resolves to SQLAlchemy internals stays blocked (sc-23186) ([@simozzy](https://github.com/simozzy)).
- `PlaidConnection` now forwards `verify_ssl`, so a UDF or notebook can turn TLS certificate verification on or off for a single connection â€” `PlaidConnection(verify_ssl=True)` â€” instead of only through the `__PLAID_VERIFY_SSL__` environment variable. Unlike `project_id`/`workspace_uuid`, it is forwarded on every path, because plaidcloud-rpc applies it after choosing a configuration source and so it overrides whichever source supplied the rest. Requires plaidcloud-rpc >= 1.12.0, the first release where `verify_ssl` actually reaches the transport (sc-23168) ([@inviscid](https://github.com/inviscid)).
- `PlaidConnection` now forwards connection configuration to `Connect` ([sc-21805](https://app.shortcut.com/plaidcloud/story/21805)): `rpc_uri`, `auth_token`, `token_provider`, `workspace_uuid` and `project_id` reach the RPC layer instead of being silently discarded, so `PlaidConnection(rpc_uri=..., auth_token=...)` configures a connection with no environment variables and no `plaid.conf`. Previously `__init__` accepted `*args`/`**kwargs` and then called `Connect.__init__(self)` with none of them, which made plaidcloud-rpc's new direct-argument API unreachable from the connection class the generated UDF boilerplate actually uses. Forwarding is by explicit name, not `**kwargs` â€” `xl_path` is consumed by `PlaidConnection` itself and `Connect` would reject it. `project_id` is passed through only alongside `rpc_uri`, since without it the environment or `plaid.conf` supplies the project. Requires plaidcloud-rpc >= 1.11.0 ([@inviscid](https://github.com/inviscid)).

- The `uuid`/`Uuid`/`UUID` cast type in transform expressions (calc fields, filters, rules) now uses the platform's canonical `GUIDHyphens` type instead of the Postgres-only `UUID`, so an inline uuid cast compiles valid SQL on every lakehouse engine (StarRocks previously dropped the cast entirely) and produces the same 36-character hyphenated form that uuid-typed columns store. Only the inline-expression cast path changes â€” column dtype configuration (`sqlalchemy_from_dtype('uuid')`) already used `GUIDHyphens` and was never affected (sc-23158) ([@inviscid](https://github.com/inviscid)).

- Security: sandbox the transform-expression engine so an authored expression can no longer execute arbitrary Python. `eval_expression`/`eval_rule` now run under a pinned restricted `__builtins__` and a strict allowlist guard: only a curated set of AST node types is permitted, no dunder/underscore attribute may be traversed (bar a single-underscore SQL-function name off `func`), `str.format`/`format_map` are blocked, and an attribute chain rooted at the raw `sqlalchemy` module is confined to type/expression helpers â€” which closes `sqlalchemy.text` raw-SQL and the module-re-export escapes (e.g. `sqlalchemy.log.logging.os.system(...)`) that a denylist could not. Legitimate SQL expressions (including `func._if`, `func.format`, and `sqlalchemy.cast`) are unaffected ([@simozzy](https://github.com/simozzy)).
- Allow `sqlalchemy.select` in the expression sandbox so scalar-subquery expressions (e.g. the Table Explorer facet's distinct-count) evaluate again, while blocking the raw-SQL splice methods it exposes (`prefix_with`/`suffix_with`/`with_hint`/`with_statement_hint`) â€” reachable on a `select(...)` result or a subquery's `.element` back-reference â€” so `select` cannot reintroduce a `text()`-style raw-SQL vector ([@simozzy](https://github.com/simozzy)).
- Add the `currency` dtype (`DECIMAL(18, 4)`, via plaidcloud-rpc >= 1.9.0 `PlaidCurrency`): accepted by the frame_join_multi validator, `import_cast` branches for HANA/Databend/StarRocks, `currency` in the expression cast vocabulary, and float64/Avro-`double` read mappings in frame_manager ([@inviscid](https://github.com/inviscid)).
- File imports load empty cells in `currency` columns as 0.0, matching numeric's behavior ([@inviscid](https://github.com/inviscid)).
- Compile keyword-based `date_add` expressions to StarRocks-compatible SQL so StarRocks dashboard filters run without parser errors.
- Add StarRocks specializations for the SQL functions the Alteryx expression converter emits, so converted expressions run on StarRocks as well as Databend. Each keeps the existing Databend/default SQL unchanged and adds only a StarRocks form: regular-expression match/extract, `to_string`/`try_to_float64` casts, `modulo`, `ord`, `today`, the date-part extractors (`to_year`/`to_month`/â€¦), the `add_*` date-add family, and unit-parameterized `date_diff` ([@inviscid](https://github.com/inviscid)).
- Close the remaining non-spatial StarRocks gaps: add StarRocks forms for `to_char` (dates via `date_format`, other values cast to `CHAR`), `to_number` (`CAST(... AS DECIMAL(38, 10))`, which also fixes the `import_cast` trailing-negatives path that previously emitted a nonexistent `to_number`), `transaction_timestamp` (`now()`), `string_to_array` (`split`), `ascii`/only-ASCII (3-arg `regexp_replace`), and `normalize_whitespace` (3-arg `regexp_replace` with RE2 `\x{â€¦}` code-point escapes). The Databend/default SQL is byte-for-byte unchanged; every StarRocks form was verified executing live ([@inviscid](https://github.com/inviscid)).
- Add a durable dialect-neutral spatial function layer (`geom_from_wkt`, `geom_point`, `geom_as_wkt`, `geom_contains`, `geom_x`, `geom_y`, `geom_within`, `geom_area`, `geom_length`, `geom_intersects`, `geom_createline`, `geom_centroid`, `geom_distance`): each renders the current Databend `ST_*` spelling on Databend/default and the verified StarRocks equivalent on StarRocks (`geom_within` swaps args into `st_contains`). Ops with no transparent StarRocks equivalent (`geom_area`, `geom_length`, `geom_intersects`, `geom_createline`, `geom_centroid`, and `geom_distance` â€” planar vs. spherical `st_distance_sphere`) raise a clear `CompileError` on StarRocks so callers degrade to the shapely executor rather than emit wrong SQL ([@inviscid](https://github.com/inviscid)).
